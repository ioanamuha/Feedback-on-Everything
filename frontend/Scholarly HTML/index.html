1. Introduction

1.1 Purpose

The purpose of this document is to present a detailed description of the Anonymous Feedback Web Application. It will illustrate the purpose and complete declaration for the development of the system. It will also explain system constraints, interface, and interactions with other external applications.

1.2 Document Conventions

The document is made with the purpose of documenting the technical means used in the creation of the website. In the document we will provide information on the architecture used, as well as details on the main functionalities and how to implement them. It is suggested that readers of the document have a technical knowledge of the web in order to understand the described implementations. For readers interested only in the technical aspects of the product, it is recommended to read sections 3 and 4 of the current document, while sections 1 and 2 are dedicated to the superficial description of the product.

1.3 Intended Audience and Reading Suggestions

This Software Requirements Specification (SRS) is tailored for a specific audience involved in the operation and interaction with the Feedback Collection Tool (FCT). Feedback Providers, the primary users, will find relevance in sections detailing the user interface and feedback mechanisms, designed to facilitate an intuitive and confidential sharing experience. Feedback Collection Initiators, who orchestrate the feedback campaigns, are directed towards system features and administrative functionalities that enable the management and curation of "things" to be evaluated. Page Administrators will prioritize content on system maintenance, user management, and data integrity, with a keen interest in the operational environment that ensures the smooth running of the FCT. Specialists and Doctors, responsible for validating responses, will find the reporting and data visualization features of particular interest, as these sections elucidate the statistical outcomes and emotional analytics derived from user feedback.

The document is structured to usher each user group to the sections most pertinent to their roles. It commences with a broad introduction, guiding Feedback Providers to interface layouts and interaction flows. Feedback Collection Initiators and Page Administrators are advised to consult the system features and external interface requirements that define their administrative capabilities. Specialists and Doctors will benefit from the latter parts of the document that elaborate on data analysis and reporting functionalities. The appendices provide supplementary material such as a glossary and TBD list, which supports all user groups in understanding the terminology and outstanding considerations. It's recommended that each user group begins with the overview sections before delving into the details most applicable to their responsibilities within the FCT ecosystem.

1.4 Product Scope

This application will provide anonymous feedback based on emotions according to the Plutchik model for various categories such as events, people, geographical places, products, services, etc.

Browsing the site allows users to access the posted feedback forms, create new anonymous (optional) ones and be able to see and download statistics generated by the responses, categorised by specialists. The generated reports will be available in HTML, CSV and JSON formats.

The product is designed, not only for normal users, but also for doctors and admins, making it possible to delete and add new forms in order to update and validate the information provided.

2. Overall Description

2.1 Product Functions

The web application is tasked with the following core functionalities:

  • Collection Mechanism: It facilitates the collection of anonymous emotional feedback from users. This mechanism is grounded in the Plutchik model of emotions.
  • Management Dashboard: This feature enables the initiator of a feedback request to oversee the items that are being evaluated. Through this dashboard, initiators can add, modify, or remove items from the list of things.
  • Response Editing: Prior to finalizing the feedback, users have the ability to edit their emotional responses, ensuring that the feedback accurately represents their sentiments.
  • Data Compilation: The system will aggregate emotional feedback responses over a designated timeframe and categorize them according to the relevant items.
  • Statistical Reporting: Post the feedback collection phase, the system will generate statistical reports reflecting emotional trends and patterns for each item evaluated, factoring in various dimensions such as user demographics, temporal aspects, and subcategories of the item.
  • Report Generation: It provides the ability to generate and export these statistics in multiple formats such as HTML for web viewing, CSV for spreadsheet analysis, and JSON for data manipulation.
  • Enhanced Data Visualization: As an additional feature, the application will incorporate sophisticated data visualization tools to present feedback statistics in a more engaging and informative manner.

    • 2.2 User Classes and Characteristics

    The software application will cater to the following user classes:

  • Feedback Initiators: This class consists of individuals or entities that require insights into the reception of various 'things'. They utilize the management dashboard extensively for setting up and monitoring feedback collection.
  • Anonymous Respondents: Users from diverse backgrounds who provide feedback without revealing their identity. The user interface for these participants will prioritize simplicity and clarity in expressing emotions.
  • Administrative Personnel: They are responsible for the maintenance and smooth operation of the web tool. Their role involves monitoring the system's health and moderating content as necessary.
  • Data Analysts: This group is focused on the data aspect, interested in obtaining detailed reports and data exports for further analysis. They require comprehensive access to the collected data and reporting features.

    • 2.3 Operating Environment

    The operating environment for the web tool encompasses:

  • Cross-Platform Accessibility: The software will be accessible via multiple web browsers including Chrome, Firefox, Opera, and Edge to ensure wide reach and usability.
  • Server Implementation: It will operate on a custom server setup, likely incorporating a modern web framework for backend operations and a robust database system for data storage.
  • Responsive Web Design: The front-end aspect of the tool will be designed to be responsive, allowing for seamless interaction across both desktop and mobile devices.
  • API Interaction: All features and data transactions will be interfaced through a REST/GraphQL API, ensuring efficient communication between the client-side and server-side components.
  • Data Integration: The tool will integrate with existing data storage and processing services to effectively manage the feedback collection and subsequent analytical processes.

    • 2.4 Assumptions and Dependencies

    2.4.1 Assumptions

    Technical Assumptions:

  • The users will have access to stable internet connections to interact with the web tool.
  • The users' devices and browsers are up-to-date and capable of supporting modern web standards necessary for the full functionality of the tool.

    • Performance Assumption: The tool assumes a certain level of performance from the server that will handle the API calls, and that it will be able to manage the expected load without significant delays or downtime.

    Security Assumption: It is presumed that the measures implemented for anonymizing feedback will be robust against attempts to de-anonymize data.

    Integration Assumption: The software components for data visualization that are to be integrated into the tool are assumed to be compatible and will not require extensive modification.

    2.4.2 Dependencies

    Third-Party Services:

  • The project is dependent on third-party services for hosting the server and the database.
  • Dependence on external APIs for some of its data processing and visualization features.

    3. External Interface Requirements

    3.1 User Interfaces

    3.1.1 Interface Components

    Interaction with the user will be through the pages that make up the site. In order to meet the necessary requirements, we have decided that the web pages implemented will be as follows: login or account creation page, home page, description page, account-specific page, statistics page, form creation page, form completion page.

    The interactions betwwen pages will be described in 4.

    3.1.2 Components Characteristics

    At the level of visual features, in this section we explain the choices made at the level of the colour scheme, placement of elements on the page and font.

    The colour scheme consists of shades of blue, with light grey tones chosen to combine the main features of the product the site serves, namely statistical elements (blue) and feedback forms (grey).

    The elements are placed in the web pages in a column, navigating the pages by scrolling. The menu is made at the top of the page to be easily accessible. The texts are accompanied by images suggestive of the content of each paragraph. Paragraphs do not span the full width of the page for ease of use. readable and to give the user an overview.

    The fonts are chosen to avoid unnecessary page load and to keep a casual and friendly in page navigation.

    3.2 Hardware Interfaces

    The web tool is predominantly dependent on two categories of hardware components for its operation: the web server and the userâ€™s device. The server is tasked with hosting the application, storing, and managing the database which contains all the information pertinent to the feedback items and the collected responses. It handles the processing of user requests and serves the web pages that display the information in a user-friendly format. Communication between the client and the server is facilitated via the HTTP/HTTPS protocol, ensuring a standard and secure exchange of data. On the client side, any device equipped with a modern web browser, including but not limited to desktops, laptops, tablets, and smartphones, is capable of accessing the web tool. This ensures that users can provide feedback from virtually any location and device, given that they have internet access.

    3.3 Software Interfaces

    3.3.1 Search engine

    In terms of software technologies, the product is designed for the most used search engines such as Chrome, Opera, Mozilla or Edge and will use their services to access the website and to communicate with the server using the HTTP protocol.

    3.3.2 Server

    The server implementation is done using Javascript as programming language, being a proprietary implementation implementation. Its role is to access the database in order to complete the accessed web pages with the information required information. The web pages will be available in template format. The specific fields of each template are filled in with data received from the server and then loaded into the browser for the user to view.

    3.3.3 Database

    The database used is Oracle Database. It will contain a table with available information as well as tags that can be applied at the time of filtering, specific to each feedback form (category). In addition to this table, there is also a table with information about the details of each form (title, description, data collection period, anonymity, owner, etc.), a table containing the validation of responses, a table of forms, a table with responses to forms and a table with encrypted information related to user login data.

    3.4 Communications Interfaces

    The HTTP protocol is the basis of communication between the site and the server. The search engine sends "HTTP Request" calls to the server, and receives "HTTP Response" signals from the server.

    Customer requests can be requests for information from the database (e.g. statistics related to form responses), or requests to write information to the database (e.g. registration of new users or forms in the database).

    The server response consists of a set of data read from the database and sent to the page that made the request for display. These data sets are strings and contain information about the requested fields in the forms or the addresses of the pictures displayed on the web pages.

    4. System Features

    4.1 Anonymous Feedback Submission

    4.1.1 Description and Priority

    This feature allows users to anonymously submit feedback for a specific item using the Plutchik model of emotions. This is a high priority feature, as it is essential to the core functionality of the product. The priority components for this feature could be rated as follows:

  • Benefit: 9 (critical to user engagement and data collection)
  • Penalty: 8 (failure to execute this feature undermines the tool's purpose)
  • Cost: 5 (moderate investment in secure and user-friendly design)
  • Risk: 4 (potential risks in handling emotional data securely)

    • 4.1.2 Stimulus/Response Sequences

  • User Action: User selects an item to provide feedback on.
  • System Response: The system presents a feedback form based on the Plutchik model.
  • User Action: User submits their emotional feedback.
  • System Response: The system confirms receipt and stores the feedback anonymously.

    • 4.1.3 Functional Requirements

  • System must present a list of items eligible for feedback.
  • Feedback form can be submitted anonymously, without storing any user identifiable information.
  • The system must frame the feedback in a category from the Plutchik wheel.
  • System must confirm the successful submission of feedback.
  • System must handle errors such as network issues or invalid input gracefully, providing clear error messages to the user.

    • 4.2 Feedback Management and Reporting

    4.2.1 Description and Priority

    This feature involves the management of feedback items and the generation of reports based on collected data. It holds a high priority as it enables the feedback initiators to analyze and interpret the emotional data effectively. Priority components are as follows:

  • Benefit: 9 (provides critical insights for feedback initiators)
  • Penalty: 7 (lack of effective reporting capabilities could reduce usability)
  • Cost: 6 (requires robust backend development)
  • Risk: 3 (low risk as it primarily deals with data already collected)

    • 4.2.2 Stimulus/Response Sequences

  • User Action: Feedback initiator logs into the management dashboard.
  • System Response: The system displays current feedback items and any associated reports.
  • User Action: Initiator requests a report based on specific criteria (time period, emotion, etc.).
  • System Response: The system generates and displays the report.

    • 4.2.3 Functional Requirements

  • System must allow initiators to view and manage feedback items.
  • System must support the generation of reports in HTML, CSV, and JSON formats.
  • Reports must be able to categorize and display data according to multiple criteria (user group, time period, etc.).
  • System must ensure data privacy and security in report generation and data handling.
  • System must provide the ability to customize report parameters to fit different analytical needs.

    • 4.3 User Account Management

    4.3.1 Description and Priority

    This feature enables the management of user accounts for feedback initiators and administrative users. It is considered a medium priority feature, essential for user authentication and role management. Priority components are as follows:

  • Benefit: 7 (ensures secure access and user-specific functionalities)
  • Penalty: 6 (lack of robust account management could compromise system security)
  • Cost: 4 (standard implementation with common security practices)
  • Risk: 5 (risks associated with data breaches and unauthorized access)

    • 4.3.2 Stimulus/Response Sequences

  • User Action: User registers or logs into their account.
  • System Response: The system validates credentials and grants access based on user roles.
  • User Action: User updates account settings or logs out.
  • System Response: The system processes the changes or ends the session securely.

    • 4.3.3 Functional Requirements

  • System must allow new users to register and existing users to log in.
  • System must authenticate users and manage session lifetimes.
  • System must provide role-based access control for different functionalities.
  • System must allow users to update their profile information securely.
  • System must log all access and changes for security auditing purposes.

    • 4.4 Data Security and Privacy

    4.4.1 Description and Priority

    This feature ensures that all data, especially sensitive feedback data, is handled and stored securely, maintaining user anonymity. It is a high priority due to its critical role in complying with privacy laws and maintaining user trust. Priority components are as follows:

  • Benefit: 9 (crucial for legal compliance and user confidence)
  • Penalty: 9 (failure to secure data could result in severe legal and reputational damage)
  • Cost: 7 (requires investment in secure infrastructure and practices)
  • Risk: 8 (high risk due to potential threats and vulnerabilities)

    • 4.4.2 Stimulus/Response Sequences

  • User Action: User submits feedback.
  • System Response: The system anonymizes and encrypts the data before storage.
  • User Action: Administrator accesses the data for reporting.
  • System Response: The system ensures that access is logged and the data remains anonymized.

    • 4.4.3 Functional Requirements

  • All user data must be anonymized at the point of entry.
  • Data must be encrypted both in transit and at rest.
  • System must implement regular security audits and vulnerability assessments.
  • Access to sensitive data must be strictly controlled and monitored.
  • System must comply with relevant data protection regulations.
